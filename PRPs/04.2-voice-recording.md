# PRP-04.2: Voice Recording for Pet Recall

## Priority: MEDIUM

## Prerequisites: PRP-04 (lost alert wizard, alert data model)

## Blocks: None (standalone enhancement)

## Problem

When a stranger finds a lost pet, the animal is often scared, disoriented, and unresponsive to unfamiliar people. Pets respond strongly to their owner's voice — hearing their name called by a familiar voice calms them and makes them approachable. No existing platform (Foundy, Petco Love Lost, Facebook groups) offers voice recordings. This is a unique differentiator that directly improves recovery success.

---

## Scope

**In scope:**

- Voice recording capture in Report Lost wizard (step 4)
- Audio file upload to Supabase Storage (max 30 seconds)
- Server-side transcoding to **M4A/AAC** (universal playback — iOS Safari, Android, all browsers)
- `voice_url` column on `pet_reports` table (already added in PRP-04)
- Voice playback on alert detail page: **both** inline `<audio>` player AND download button
- Download button as primary CTA (works offline, avoids WebView codec issues)
- Explicit consent checkbox (PDPA — voice is PII)
- Audio visualization during recording (simple waveform or timer)

**Out of scope:**

- Voice transcription or analysis
- Audio editing / trimming
- Voice-to-text for accessibility (future)
- Complex audio effects or noise reduction

## Tasks

### 4.2.1 Recording UI Component

- [ ] Create `components/voice-recorder.tsx` — client component
- [ ] Uses `MediaRecorder` Web API (widely supported in LINE LIFF WebView)
- [ ] Max duration: 30 seconds with countdown timer
- [ ] Visual feedback: recording indicator (red dot + timer), waveform or simple pulse animation
- [ ] Controls: Record / Stop / Play back / Re-record / Delete
- [ ] Consent checkbox: "I consent to uploading my voice recording. It will be playable by anyone viewing this alert." (PDPA requirement)
- [ ] Output: audio blob (WebM/Opus from browser recording)

### 4.2.2 Audio Upload & Transcoding

- [ ] Upload raw audio blob to Supabase Storage bucket `voice-recordings/`
- [ ] **Server-side transcode to M4A/AAC** via API route (`app/api/voice/route.ts`):
  - Accept raw WebM/Opus upload from browser
  - Transcode to M4A/AAC using `ffmpeg` (via `fluent-ffmpeg` or `@ffmpeg/ffmpeg` WASM)
  - M4A/AAC plays natively on iOS Safari, Android, all browsers — no codec issues
  - Store transcoded M4A as the final file, delete raw WebM
- [ ] File naming: `{alert_id}_{timestamp}.m4a`
- [ ] Max file size: 2MB (30s at reasonable quality)
- [ ] Update `pet_reports.voice_url` with transcoded M4A URL after upload
- [ ] Storage bucket policy: public read, authenticated write
- [ ] Set `Content-Disposition: attachment` header for download endpoint

> **Why transcode:** WebM/Opus (browser default) doesn't play on iOS Safari. M4A/AAC is the universal format that works everywhere — inline playback AND download-and-play on native music app.

### 4.2.3 Playback & Download on Detail Page

- [ ] Add voice section on `app/post/[id]/page.tsx` with **two options**:
  1. **Download button (primary CTA)**: "📥 ดาวน์โหลดเสียงเจ้าของ" (Download owner's voice)
     - Downloads `.m4a` file to device
     - Stranger plays from native music/file app — works offline, max volume, no browser issues
     - Uses `<a href={voice_url} download="[pet_name]-owner-voice.m4a">` or API route with `Content-Disposition: attachment`
  2. **Inline player (secondary)**: simple `<audio>` element with play/pause
     - For quick preview before downloading
     - May not work in all LIFF WebViews — download is the reliable fallback
- [ ] Label: "🔊 เสียงเจ้าของเรียกน้อง [pet name]" (Owner's voice calling [pet name])
- [ ] Instruction text: "หากพบน้อง ให้กดดาวน์โหลดแล้วเปิดเสียงใกล้ๆ น้อง เสียงเจ้าของจะช่วยให้น้องสงบลง" (If you find the pet, download and play near them. Owner's voice will calm them.)
- [ ] Fallback: hide entire section if no voice_url

> **Why download-first:** LIFF WebView `<audio>` playback is unreliable across devices. A downloaded M4A file plays natively on every phone's default player — full volume, works offline, no codec issues. The stranger is likely outdoors looking for the pet — they need the audio file on their phone, not a browser tab open.
